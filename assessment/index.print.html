Competency-Based Assessment

How the program assesses against the structure. Competencies belong to the program, not to individual courses; courses are evidence-generating contexts. Grades and competency status run as parallel tracks.

Subsections of Competency-Based Assessment

The Competency Model

Competency assessment in this program operates at the program level, not the course level. Competencies belong to the degree, not to individual courses; courses are evidence-generating contexts. A student’s competency record is maintained across their entire time in the program through the Crystalis assessment platform.

How Evidence Accumulates

Assessment is continuous and based on a preponderance of evidence — no single event determines a competency level. Evidence accumulates from multiple courses and multiple evidence types, and Crystalis aggregates it into a frequently-updated assessment score with a confidence interval visible to both faculty and the individual student, along with a reviewable record of contributing evidence, claims, assessments, and collected artifacts (such as the code that was assessed).

Evidence Tiers

Two evidence tiers feed the competency record:

Tier 1 — Automated Evidence

Crystalis collects telemetry automatically from instrumented learning environments:

  • E-textbook and video engagement
  • Embedded H5P exercises (scaffolded code workouts and interactive worksheets)
  • Git logs and commit history
  • Time logs
  • AI chat prompt and response logs
  • Near real-time keystroke and event logging from instrumented IDEs

Automatically collected evidence is aggregated into claims that faculty review and approve. Automated evidence can contribute up to the Application level.

Tier 2 — Direct Observable Assessments

Faculty evaluate directly observable student work through LTI 3.0-integrated rubrics. These assessments are designed to expose student thinking and comprehension in contexts that automated evidence cannot capture:

  • Code readings and code reviews
  • Debugging challenges
  • Design briefs and evaluation reports
  • Performance assessments and reflection assessments
  • Integrative projects

Direct assessments can reach Independence, Adaptation, and Leadership.

The Competency Level Scale

Level Description
Exposure Recognizes concepts and vocabulary
Application Uses the skill in structured tasks
Independence Uses the skill without scaffolding
Adaptation Applies the skill in novel contexts
Leadership Teaches, critiques, and improves practice

Leadership is not limited to course contexts. Students can demonstrate Leadership through undergraduate TA/LA roles, club leadership, peer mentoring, conference presentations, and other professional engagement outside the classroom.

Parallel Tracks: Grades and Competencies

Grades live in administrative course containers and reflect a student’s learning engagement within that course — they are not solely a measure of proficiency. Competency levels live in the program-level Crystalis record and reflect demonstrated capability across multiple contexts.

These are parallel tracks, not redundant ones. A student may pass a course without yet demonstrating independence on a competency; a student may demonstrate a competency through non-course evidence before the relevant course has formally been completed.

Graduation Artifacts

Students graduate with two records:

  1. Traditional transcript — course-by-course grade record
  2. Final competency report — program-level competency assessment with evidence citations and a reviewable evidence trail

Students may prepare an employer-facing version of their competency report through Crystalis: a curated presentation of their competency levels with selected supporting evidence artifacts (code samples, assessments, project work) chosen by the student.

Target Level

The program’s aspiration is that every student exits at the Adaptation level or above across all competencies. Due to institutional constraints, competency levels are maintained as a parallel record independent of graduation requirements.

Competencies

The competency domains are:

  1. Computational Reasoning
  2. Software Development
  3. Data and Information
  4. Systems and Infrastructure
  5. Human-Centered Practice
  6. Professional Practice

Subsections of Competencies

Computational Reasoning

Analyze Computational Systems

I can explain the behavior of computational systems by investigating their components, interactions, and execution.

I analyze software and computing systems to determine how they operate, identify sources of behavior, and predict outcomes under varying conditions.

Primary evidence contexts:

Block Course Work
B1 CS-103 Memory model, process model, boolean/number representation — first named encounter with how a running system works
B2 CS-105 Code Archaeology (signature) — structured investigation of unfamiliar code to explain behavior
B2 CS-106 Browser as mini-OS (event loop, origin isolation, web workers, postMessage) — analyzing a concrete computational system
B3 CS-107 Analyzing component interactions through the encapsulation/contract lens
B4 CS-111 Client-server dataflow; API as system structure
B8 CS-210 Performance analysis on a real deployed bottleneck; OS/concurrency model analysis

Assessment types by level:

Level Assessment type
Application Code reading — trace execution and explain behavior with guided prompts
Independence Code Archaeology — unscaffolded, unfamiliar codebase, student explains without hints
Adaptation Performance investigation (CS-210) — analyze real system behavior where empirical results diverge from theory
Leadership TA-led code reading sessions; mentoring peers through code archaeology in LA role

Recurring type: Code Archaeology — deepens from single-module familiar code (B2) through production system diagnosis (B8). Also: System Analysis Task for behavior traced through operational signals rather than code.

Signature Assessments

Signature assessments are the five formal, high-stakes assessment events distributed across the program. Each is a culminating instance of a recurring assessment type — it represents that type at its most consequential, typically involving real artifacts, peer or faculty audience, and no scaffolding.

Code Archaeology Milestone

Block 2 — CS-105 Code Reading & Repair

Students investigate an unfamiliar, moderately complex codebase — written in languages they know — to explain its behavior, identify sources of unexpected behavior, and propose a repair. This is the first unscaffolded instance of the Code Archaeology recurring type.

Code Archaeology does not end at Block 2. It recurs and deepens throughout the program (see Recurring Assessment Types), but the Block 2 milestone is the first formal, evaluated instance.

Evidence generated: Analyze Computational Systems; Maintain Software Systems

Data Investigation

Block 5 — CS-202 Database Design

Students identify a question they want to investigate, acquire or are given a novel dataset, and conduct a complete investigation: schema design, query strategy, statistical summarization, and interpretation. The investigation is documented and presented.

This is the formal instance of the Data Investigation recurring type and anchors the Store block’s evidence for the Data and Information domain.

Evidence generated: Investigate Questions with Data; Manage Data Resources; Represent Information

Design Review

Block 7 — CS-207 Graphs & Network Algorithms

Students present a system design — algorithm selection, data model choice, and architectural tradeoffs — to a panel of peers and faculty and defend their choices under critique. The defense must address correctness, performance, human-centered concerns, and ethical implications.

Design Review is the highest-stakes instance of both the Design Brief and the Technical Explanation Task recurring types. It recurs in a lighter form at Block 8.

Evidence generated: Design Software Systems; Develop Abstractions; Evaluate Algorithmic Solutions; Communicate Technical Information; Evaluate Ethical Implications

Team Software Project

Block 6 (CS-206) through Block 8 (CS-211)

Students collaborate on a shared software product across three blocks, taking it from requirements through implementation, testing, deployment, and a blameless postmortem. The project carries a responsibility retrospective at Block 6 and a final postmortem at Block 8.

This is the primary evidence context for the Professional Practice domain and generates sustained evidence for collaborative, implementation, maintenance, and verification competencies.

Evidence generated: Collaborate on Software Teams; Implement Software Solutions; Verify Software Quality; Maintain Software Systems; Evaluate Ethical Implications; Reflect on Professional Growth

System Integration Project

Block 8 — CS-211 Human-Centered Design & Validation

Students design, build, deploy, and validate a multi-component system for real users. The project includes formal requirements gathering, acceptance testing with actual users, and a blameless postmortem. Students prepare an employer-facing competency report drawing on their full Crystalis evidence record.

This is the program’s culminating assessment event and generates evidence across all six competency domains.

Evidence generated: All domains; final evidence anchor for Adaptation-level claims
